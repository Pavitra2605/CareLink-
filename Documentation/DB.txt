# 🎯 STEP 1: WHO WILL USE CARELINK?

We classify users based on:

* Medical responsibility
* System access level
* Operational control
* Data sensitivity

---

# 👥 USER TYPES (PRIMARY ROLES)

## 1️⃣ PATIENT (Core End User)

### Who?

* Rural individuals
* Elderly users
* Chronic care patients
* Family members using shared device

### Access Level

* Symptom check
* Emergency SOS
* Consultation request
* View own records
* Upload reports

### Cannot:

* View other patients
* Modify triage rules
* Access doctor data

---

## 2️⃣ DOCTOR

### Who?

* Licensed medical professionals
* Remote teleconsult doctors

### Access Level

* View assigned patients
* Accept consultations
* Write consultation notes
* Upload prescriptions
* Mark case status

### Cannot:

* Modify triage rules
* View all system data
* Access admin panel

---

## 3️⃣ COMMUNITY HEALTH WORKER (CHW)

### Who?

* Village-level health assistants
* ASHA workers
* NGO field staff

### Access Level

* Register patients
* Assist with triage
* Trigger emergency on behalf of patient
* View limited patient history

### Restricted:

* Cannot access unrelated patients
* Cannot prescribe

---

## 4️⃣ ADMIN

### Who?

* System administrators
* Operations team

### Access Level

* Manage users
* View analytics
* Monitor emergencies
* Assign doctors
* Manage service availability

### Cannot:

* Alter medical notes content
* Diagnose

---

## 5️⃣ SUPER ADMIN (Infrastructure Control)

### Who?

* Platform technical owner

### Access Level

* Modify system configuration
* Deploy triage rule updates
* Manage RabbitMQ & services
* Audit logs

This role is NOT available in mobile app.

---

# 🏥 ROLE SUMMARY TABLE

| Role        | Platform | Scope             | Data Access       |
| ----------- | -------- | ----------------- | ----------------- |
| PATIENT     | Mobile   | Self-only         | Own records       |
| DOCTOR      | Web      | Assigned cases    | Assigned patients |
| CHW         | Mobile   | Assisted patients | Linked patients   |
| ADMIN       | Web      | Operational       | Metadata          |
| SUPER_ADMIN | Backend  | System-level      | All logs          |

# 🗄 STEP 2: DATABASE ROLE ARCHITECTURE (PostgreSQL)

We use RBAC with separate role table.

---

## 🔹 TABLE: roles

```
id (uuid)
name (PATIENT | DOCTOR | CHW | ADMIN | SUPER_ADMIN)
description
created_at
```

---

## 🔹 TABLE: users

```
id (uuid)
phone_number (unique)
full_name
role_id (FK roles.id)
is_active (boolean)
created_at
updated_at
```

---

## 🔹 TABLE: user_profiles

Separate profile table for extensibility.

```
id
user_id (FK users.id)
gender
date_of_birth
blood_group
address
emergency_contact
created_at
updated_at
```

---

## 🔹 TABLE: doctor_profiles

```
id
user_id (FK users.id)
license_number
specialization
years_experience
availability_status
created_at
```

---

## 🔹 TABLE: chw_profiles

```
id
user_id
assigned_region
organization
```

---

## 🔹 TABLE: admin_profiles

```
id
user_id
permission_level
```

---

# 🔐 STEP 3: PERMISSION MODEL (Granular)

Instead of only role-based,
we implement permission mapping.

---

## 🔹 TABLE: permissions

```
id
name
description
```

Examples:

* VIEW_OWN_RECORD
* VIEW_ASSIGNED_PATIENT
* CREATE_CONSULTATION
* TRIGGER_EMERGENCY
* MANAGE_USERS
* VIEW_ANALYTICS
* UPDATE_TRIAGE_RULES

---

## 🔹 TABLE: role_permissions

```
role_id
permission_id
```

This allows:

* Future flexibility
* Enterprise-level scalability
* Easy compliance auditing

# 🏗 DATABASE DESIGN STRATEGY

Separation of concerns:

* PostgreSQL → Structured user & operational data
* MongoDB → Medical records (unstructured)
* Redis → Session cache
* RabbitMQ → Event-based permissions enforcement

---

# 🔒 SECURITY DESIGN

Each API route checks:

1. JWT
2. Role
3. Permission
4. Resource ownership

Example:

Patient accessing records:

```
if (user.id === record.patient_id)
```

Doctor accessing:

```
if (doctor assigned to consultation)
```
